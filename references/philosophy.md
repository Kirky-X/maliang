# DESIGN.md 设计哲学

三条原则指导 DESIGN.md 的撰写。设计真正存活于叙述之中,而非 token 之中。在撰写任何 `## Overview`、`## Colors` 或 `## Do's and Don'ts` 章节之前,先读本文。

## 1. 叙述优先,token 作为上下文

生成设计的质量,较少取决于数值的精确度,更多取决于意图被描述得有多清晰。Token(`colors`、`typography`、`spacing`、……)是 agent 与叙述一并阅读的上下文,它们不是渲染指令。一份 DESIGN.md 若 hex 值完美但叙述模糊,产出的是平庸之作;一份 DESIGN.md 若叙述鲜活而数值近似,产出的是独特之作。

**如何应用:** 把最多的撰写精力花在每个 `##` 章节下的叙述上。让叙述解释一个 token _为什么_ 存在、_何时_ 使用,而不仅仅是命名它。

## 2. 一个具体参照胜过一串形容词

"一份老牌大学传统下的 1970 年代研究生讲义" 唤起一个完整的世界——单一墨色、宽裕的边距、阅读字号的衬线体、毫无装饰。这一句话承载的有用信息多过一打度量值,因为它承载了数值背后的推理。

"现代、干净、可信、高端" 唤不起任何具体之物。形容词描述的是一个 _区域_;具体参照描述的是一个 _点_。模型会用通用中心去填满那个区域。

**如何应用:** 在 `## Overview` 中,把设计锚定到一个具体的参照物——一个时代、一份出版物、一款产品、一种物理材质。用一处能唤起想象的参照替代堆砌的形容词。

## 3. 负向约束定义性格

你省去什么,就定义了什么风格。一个具体的参照会自动携带它的限制:模型知道讲义不会发光、不会用渐变,就像说"狗"就告诉模型狗不会喵喵叫。

一份刻意的 `## Do's and Don'ts` 列表是有用的。一份冗长散漫的 don'ts 列表,往往是描述太模糊、无法隐式承载约束的信号。强参照 _加上_ 一份刻意的 do's/don'ts 列表,才是最佳点。

**如何应用:** 把 `## Do's and Don'ts` 写成刻意的、果断的约束。如果 don'ts 列表变长,修复办法是让 `## Overview` 的参照更具体,而不是再加更多 don'ts。

## 不鼓励 token 级要求

DESIGN.md 记录设计的 _本质_;它不重做语言和工具已经完成的工作。通常不要在规范中强加 token 级要求。把 token 保留为叙述所引用的上下文(例如 `**Ink** {colors.ink} is graphite-warm…`)。
